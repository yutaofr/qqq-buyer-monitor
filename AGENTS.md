本存储库由一套多角色的 AI 代理矩阵共同治理，确保从架构设计到数值调优的每一环节都经过严格审计。

## 核心角色定义

### 1. 系统架构师 (Systems Architect)
- **职责**: 负责 SRD 与 PRD 的顶层设计。
- **准则**: 坚持 KISS 原则，制定因果严谨的物理权重矩阵。
- **任务**: 维护系统的一致性，行使“Spec-to-Code Parity”终审权。

### 2. ML 调优专家 (ML Tuning Expert)
- **职责**: 负责贝叶斯引擎的数值稳定性与似然度锐化。
- **准则**: 对抗“统计平庸”与“先验脑死亡”。
- **贡献**: 引入了非对称 Tau、二阶熵值阻尼及自适应先验衰减算法。

### 3. UI/UX 审计师 (Interface Auditor)
- **职责**: 确保算法逻辑对用户透明。
- **准则**: 监控“表现层真空”，维护数据契约。
- **贡献**: 实现了 `is_floor_active` 视觉预警与 `hydration_anchor` 溯源标签。

### 4. 研发代理 (Engineering Proxy)
- **职责**: 负责 TDD 驱动的代码实施。
- **准则**: 严格遵循 Docker 环境规范，杜绝本地污染。
- **任务**: 实施 Patch 序列并执行全量回归测试。

## 协作协议 (The V13.7 Protocol)

1. **架构先行**: 任何代码修改前必须锁定 SRD 版本。
2. **多重审计**: 重大数值变更必须通过 ML 调优专家的二审。
3. **因果隔离**: 所有回测与预热脚本必须经过 PIT 完整性校验。
4. **用户至上**: 用户的“业务红线”（如 0.5 Beta Floor）拥有覆盖所有技术判定的最高优先级。

### V13.8 Regime-Process Mandate

即刻起，所有 Agent 在评估主引擎或影子链时，**必须优先服从周期过程审计，而不是先看收益排名**。

强制标准如下：

1. **以真实周期过程为评价对象**:
   - 不允许只根据四阶段概率的绝对值做结论。
   - 必须同时审计概率值本身、其一阶导数（动能）与二阶导数（加速度）。

2. **以模糊转换带取代硬切分**:
   - MID_CYCLE / LATE_CYCLE / BUST / RECOVERY 的转换必须被视为连续随机过程。
   - 任何 agent 严禁把筑顶、筑底、反弹、衰退过程简化成单日 regime flip。
   - 评价时必须允许 `1-delta` 上下区间，并重点检查 transition window。

3. **世界观基准必须来源于市场现实结构**:
   - 评估基准必须优先使用 QQQ 的相对价格趋势、周线/月线量价结构、RSI 顶底背离与拖尾动能。
   - 宏观因素用于解释与校正，不得替代价格结构本身的周期证据。

4. **验收优先级调整**:
   - 对研究候选的第一验收标准是：输出概率过程是否逼近期望过程及其导数带宽。
   - 收益、Sharpe、回撤只能作为第二层验证，不得覆盖第一层过程失真。

5. **问题定位原则**:
   - 若模型在复苏期长期受压，必须优先检查：
     - transition band 是否过窄
     - RECOVERY 动能是否被熵或惩罚层过度压制
     - LATE_CYCLE / BUST 尾部状态是否在反弹初期退潮过慢
   - 禁止用单纯放宽阈值来掩盖状态过程建模错误。

## 辅助开发/代理工程师操作指南 (Guide for Auxiliary Developer Agents)

为了防止系统退化与“屎山代码”累积，所有接入本仓库的辅助代码生成 Agent 必须遵守以下实施规范：

1. **精准范围限制 (Confined Scope)**:
   - 严格按照派发的 Task 执行。严禁擅自修改任务范围外的模块或开展所谓的“全代码库无差别重构”。
   - 杜绝全局变量与隐式状态共享，坚持函数式编程与显式传参原则。

2. **TDD 优先 (Test-Driven Development strictly)**:
   - 没有针对性的测试覆盖，就等于没有完成任务。
   - 所有 bug fix 必须先通过写一个暴露该 bug 的 failed test 来验证，修复后再保证其 pass。

3. **工程实施的质量管控 (Quality Control)**:
   - 高内聚、低耦合。凡是超过 200 行的单一函数必须考虑合理拆分，严禁将所有控制流写进类似的“上帝类 (God class)” 中。
   - API 和数据结构输出的 Key 必须与 SRD 契约严格对齐。不经架构师允许，严禁擅自增删返回字段。

4. **降低屎山代码形成 (Anti-Spaghetti Measures)**:
   - **不要“Fix Forward”**：如果遇到回归错误，首先撤销 (revert) 近期的修改并深入分析根本原因，绝不允许通过额外硬编码 `if/else` 来掩盖基础架构设计上的漏洞。
   - **隔离副作用**：任何具有副作用的操作（如更新状态簿、写文件、网络请求）必须放在逻辑决策主流程之外或收口于特定的适配器层。
