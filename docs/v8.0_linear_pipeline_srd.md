# SRD: v8.0 线性指令链资产配置系统

> 版本: v8.0
> 状态: Draft
> 日期: 2026-03-25
> 适用范围: `qqq-monitor` 生产重构
> 前置版本: v7.0 (`docs/v7.0_industrial_allocation_srd.md`)
> 改造来源: `docs/plans/2026-03-25-v7-tier0-risk-deployment-separation-design.md`

## 1. 文档目的

本 SRD 定义 `qqq-monitor` v8.0 的目标系统。

v8.0 在 v7.0 双控制器架构基础上，解决三个在生产运行中暴露的结构性缺陷：

1. **Tier-0 宏观判定未接入决策约束链**，导致宏观结构恶化时系统仍可能输出 `RISK_NEUTRAL`。Tier-0 对存量 beta 推荐是硬约束，对增量资金入场是减速器（降低速度但不一票否决，保留左侧入场机会）。
2. **存量 beta 推荐与增量入场节奏推荐混淆**，系统越界计算具体金额，实际上系统不应该管理用户的钱包和账户，只负责信号层面的推荐。
3. **缺少 QQQ 和 QLD 的第一性原理分析**，两个资产的风险因子、适用场景和决策逻辑本质不同，不能混在同一个 beta 框架里一刀切。

v8.0 的核心目标用一句话概括：

> 系统只做两件事：根据信号推荐存量目标 beta，以及推荐增量资金的入场时机和节奏。不管钱，不下单，不算金额。

## 2. v7.0 遗留问题诊断

### 2.1 Tier-0 断链

**现状代码 (`risk_controller.py`)：**

`decide_risk_state()` 函数签名中没有 `tier0_regime` 参数。Tier-0 的 `StructuralRegime` 判定结果仅在 aggregator 的叙事层和日志层出现，没有参与 `RiskDecision` 的构建过程。

**后果：**

```
Tier-0 = RICH_TIGHTENING (资金成本昂贵，流动性收缩)
↓ 被忽略
Risk Controller = RISK_NEUTRAL (微观指标还行)
↓
Deployment Controller = DEPLOY_FAST (允许加速部署)
```

宏观最坏结构被提示出来了，但风控仍然给出中性甚至乐观的许可。

### 2.2 系统越界：推荐引擎不应该管理钱包

**现状代码 (`execution_policy.py`)：**

`build_execution_actions()` 计算 `deploy_cash_amount`（具体美元金额），`RebalanceAction` 输出具体仓位百分比。系统在做推荐 beta 之外的事情——它在替用户算账、计算调仓金额，这超出了一个信号推荐系统的职责边界。

**后果：**

- 系统输出看起来像是在管理用户的账户
- 用户容易把推荐误读为执行指令
- `DEPLOY_FAST + $0` 这种输出制造了语义混乱——到底是想买还是不想买？
- 系统的真正价值是信号和 beta 推荐，不是计算具体买多少钱的 QQQ

### 2.3 输出语义模糊

当前输出分成"资产配置风险管理"和"增量资金买入时机决策"两个区块，但：

- "资产配置风险管理"里混杂了仓位比例细节，更适合放在存量再平衡区块。
- 缺少对"这一切都是建议/推荐，不是自动执行"的显式声明。
- 用户看到 `DEPLOY_FAST + $0` 容易理解为"系统想买但没执行"，实际语义应该是"部署通道为空"。

### 2.4 v6 遗留状态机冗余

`allocation_search.py` 仍然依赖 v6 的 `AllocationState`（如 `SLOW_ACCUMULATE`、`PAUSE_CHASING`），这些状态词汇与 v7 的 `RiskState` / `DeploymentState` 形成平行控制面，增加了语义冲突和维护成本。v8.0 需要切断这条依赖。

## 3. 产品目标

### 3.1 系统定位

系统是一个**信号推荐引擎**，不是投资组合管理器。它的输出边界是：

- ✅ 推荐当前环境下的组合级目标 beta（`portfolio_beta = qqq_pct × 1.0 + qld_pct × 2.0`，现金占比越高，组合 beta 越低）
- ✅ 推荐增量资金的入场时机和节奏
- ❌ 不计算具体调仓金额
- ❌ 不展示偏离程度（用户自己看得到差距）
- ❌ 不管理用户的钱包或账户
- ❌ 不自动执行任何交易

### 3.2 主目标

- 在可承受风险预算内推荐长期复利导向的目标 beta。
- 宏观环境恶化时，通过 Tier-0 硬约束降低存量 beta 推荐上限。
- 宏观环境恶化时，通过 Tier-0 软约束降低增量资金入场速度，但保留左侧入场机会。
- 对 QQQ 和 QLD 分别分析各自的风险因子，但最终输出的目标 beta 是组合级的（QQQ + QLD + Cash 整体）。
- 所有输出明确标注为"推荐/建议"。

### 3.3 量化目标

- `G1`: 组合级最大回撤预算长期控制在设计阈值内，默认预算 `30%`。
- `G2`: 当 Tier-0 判定为 `CRISIS` 或 `RICH_TIGHTENING` 时，`Risk Controller` 的存量 beta 推荐必须直接反映宏观压制。
- `G3`: 增量资金入场推荐在 `RICH_TIGHTENING` 下降速但不熄火，当出现高质量超跌时仍允许 `DEPLOY_SLOW` 或更高。
- `G4`: 风险评估、存量 beta 推荐、增量入场推荐三个语义在输出中物理隔离。
- `G5`: 所有生产决策可通过 JSON / SQLite / 报告完整回放。

## 4. 非目标

- 不追求日内交易。
- 不追求市场中性。
- 不追求"最准抄底点"叙事。
- 不允许使用不可验证的模拟因子来包装收益来源。
- 不允许在运行时临时发明新候选权重。
- **不管理用户账户**：不计算具体金额，不模拟交易执行，不替用户做钱包层面的决策。
- **不一票否决左侧入场**：即使宏观环境偏紧，只要有高质量超跌信号，增量部署通道不能完全关死。

## 5. 设计原则

- `P1 单向信息流`: 决策链必须严格按 Tier-0 → Risk → Search → Recommend 单向流转，下游不可逆转上游结论。
- `P2 宏观分级约束`: Tier-0 对存量 beta 推荐是硬约束（不可绕过），对增量资金入场是软约束（减速但保留左侧机会）。
- `P3 存量/增量物理隔离`: 存量 beta 推荐和增量入场节奏推荐是两个独立引擎，各自有独立的触发条件和输出逻辑。
- `P4 硬约束优先`: 风险预算、宏观约束、数据可用性优先于收益排序。
- `P5 无伪造数据`: 缺失即缺失，占位值不得进入生产决策主路径。
- `P6 低频执行`: 允许高频审计，不允许无条件高频调仓。
- `P7 可回放`: 任一日的结果都必须能复建输入、状态、候选、筛选和最终输出。
- `P8 推荐而非执行`: 系统只输出 beta 推荐和入场节奏推荐，不计算金额，不管理账户。
- `P9 QQQ/QLD 独立建模`: 两个资产的风险因子本质不同，必须分别分析和建模。

## 6. v8.0 核心架构：单向线性流水线

### 6.1 四级决策链（The Linear Pipeline）

v7.0 的网状逻辑被替换为绝对单向流水线：

```
[1] 宏观判定 (Tier-0 Regime)
      │
      │  输出: StructuralRegime
      │  对存量: 硬约束 (beta 天花板)
      │  对增量: 软约束 (降速不熄火)
      ▼
[2] 风险定界 (Risk Controller)
      │
      │  接收: tier0_regime (硬输入)
      │  输出: RiskDecision (目标 beta 推荐)
      ▼
[3] 战略寻优 (Allocation Search)   ← 从认证候选库中选择
      │
      │  输入: max_beta_ceiling + max_drawdown_budget
      │  输出: 推荐配比 (QQQ:QLD:Cash)
      ▼
[4] 双轨推荐 (Dual Recommendation)
      ├── [4a] 存量 beta 推荐：推荐目标 beta 和偏离程度
      └── [4b] 增量入场推荐：推荐入场时机和节奏
```

### 6.2 Tier-0 宏观判定层

Tier-0 是系统的最高权威，但它对不同下游模块的约束力度不同：

- **对存量 beta 推荐：硬约束**。Tier-0 直接压低 Risk Controller 的 `target_exposure_ceiling`，微观指标不能绕过。
- **对增量入场节奏：软约束**。Tier-0 降低默认部署速度，但如果 Class B 数据显示高质量超跌/恐慌信号，仍允许提速。这保留了左侧入场的窗口。

#### 输入

- `credit_spread` (Class A)
- `equity_risk_premium` (Class A)

#### 输出：`StructuralRegime`

| Regime | 语义 | 对存量 beta 推荐（硬约束） | 对增量入场推荐（软约束） |
| :--- | :--- | :--- | :--- |
| `EUPHORIC` | 信用极宽松、ERP 极高 | 允许最大化进攻 | 允许 `DEPLOY_FAST` |
| `NEUTRAL` | 宏观中性 | 正常运行 | 允许 `DEPLOY_FAST` |
| `RICH_TIGHTENING` | 资金成本昂贵或ERP过低 | 强制 `ceiling ≤ 0.30` | 默认降至 `DEPLOY_SLOW`，但若 capitulation 信号强烈允许提至 `DEPLOY_BASE` |
| `TRANSITION_STRESS` | 信用或ERP单边恶化 | 强制至少 `RISK_REDUCED` | 默认 `DEPLOY_SLOW`，若 capitulation 信号强烈允许提至 `DEPLOY_BASE` |
| `CRISIS` | 信用 ≥ 500bps 或 ERP < 1.0% | 强制 `RISK_EXIT`，`ceiling = 0.0` | 强制 `DEPLOY_PAUSE` |

### 6.3 Risk Controller（风险定界层）

在 v8.0 中，`decide_risk_state()` 的函数签名必须显式接收 `tier0_regime` 参数。

#### 新签名

```python
def decide_risk_state(
    snapshot: FeatureSnapshot,
    portfolio: CurrentPortfolioState,
    tier0_regime: str,             # <-- v8.0 新增：硬输入
    drawdown_budget: float = 0.30,
) -> RiskDecision:
```

#### 判定优先级（从高到低）

1. **Tier-0 绝对否决**（最高优先级）
   - `CRISIS` → `RISK_EXIT`，`target_exposure_ceiling = 0.0`
   - `RICH_TIGHTENING` → `RISK_REDUCED`，`target_exposure_ceiling ≤ 0.30`
   - `TRANSITION_STRESS` → 至少 `RISK_REDUCED`
2. **Class A 数据缺失降级**
3. **回撤预算硬约束**
4. **信用 / 流动性 / 资金压力共振**
5. **单边恶化**
6. **清洁环境**

#### 输出：`RiskDecision`

- `risk_state: RiskState`
- `target_exposure_ceiling: float`
- `target_cash_floor: float`
- `tier0_applied: bool` — 标记是否因 Tier-0 否决而修正了结果
- `reasons: tuple[dict, ...]`

### 6.4 Allocation Search（战略寻优层）

v8.0 的 `find_best_allocation()` 不再接收 v6 的 `AllocationState` 状态词汇，只接收两个数学约束：

1. `max_beta_ceiling`（由 Risk Controller 给出）
2. `max_drawdown_budget`（AC-5 容忍度，默认 0.30）

在认证候选库内搜索满足约束的最优配比。如果没有任何候选满足约束，返回 `100% Cash`。

### 6.5 双轨推荐层

#### 6.5.1 存量 beta 推荐

**系统只输出组合级目标 beta，不计算调仓金额。**

输出内容：

- `target_beta`：当前环境下推荐的组合级 `effective_exposure`
- `recommended_allocation`：推荐的 QQQ:QLD:Cash 配比（来自认证候选）
- `should_adjust: bool`：当前持仓是否偏离目标足够大，建议用户关注

触发"建议调整"的条件（OR 逻辑）：

- `risk_state` 变化
- `exposure_drift > exposure_band`（默认 0.03）
- 风险预算触发硬门槛

**系统不输出的内容：**

- 具体买卖多少股或多少美元
- 基于用户账户余额的任何计算

#### 6.5.2 增量入场节奏推荐

- **职责**：推荐新增资金（场外新注入的钱）的入场时机和速度。
- **Tier-0 软约束**：宏观偏紧时降低默认速度，但当 Class B 指标（`capitulation_score`）显示高质量超跌时，仍允许提速。这确保在 `RICH_TIGHTENING` 下也能捕捉左侧入场机会。
- **当 `available_new_cash = 0` 时**：输出 `DEPLOY_IDLE`（v8.0 新增），表示"部署通道为空"。
- **部署速度规则**：
   - `DEPLOY_FAST`：恐慌/超跌确认，加速入场。
   - `DEPLOY_BASE`：正常 DCA 节奏。
   - `DEPLOY_SLOW`：减速入场，市场承压但未形成高质量超跌。
   - `DEPLOY_PAUSE`：仅在 `CRISIS` 时强制暂停。
   - `DEPLOY_RECOVER`：从暂停恢复，平滑提速。
   - `DEPLOY_IDLE`：**v8.0 新增**，无新增资金，部署通道为空。

## 7. Tier-0 投影规则

### 7.1 `StructuralRegime` → `RiskDecision` 映射

| Tier-0 Regime | Risk Controller 约束 | 最高允许 `RiskState` | `target_exposure_ceiling` |
| :--- | :--- | :--- | :--- |
| `CRISIS` | 强制 `RISK_EXIT` | `RISK_EXIT` | `0.00` |
| `TRANSITION_STRESS` | 强制至少 `RISK_DEFENSE` | `RISK_DEFENSE` | `0.50` |
| `RICH_TIGHTENING` | 强制至少 `RISK_REDUCED` | `RISK_REDUCED` | `0.30` |
| `NEUTRAL` | 不施加额外约束 | — | — |
| `EUPHORIC` | 允许最大进攻 | `RISK_ON` | `1.00` |

### 7.2 对增量入场推荐的软约束

Tier-0 设定默认速度上限，但 Class B 数据（`capitulation_score` 达标）可以突破默认限制，保留左侧入场窗口。

| Tier-0 Regime | 默认速度上限 | 超跌信号突破后上限 |
| :--- | :--- | :--- |
| `CRISIS` | `DEPLOY_PAUSE` | `DEPLOY_PAUSE`（不可突破） |
| `TRANSITION_STRESS` | `DEPLOY_SLOW` | `DEPLOY_BASE` |
| `RICH_TIGHTENING` | `DEPLOY_SLOW` | `DEPLOY_BASE` |
| `NEUTRAL` | `DEPLOY_FAST` | `DEPLOY_FAST` |
| `EUPHORIC` | `DEPLOY_FAST` | `DEPLOY_FAST` |

## 8. 数据分级与使用协议

v8.0 继承 v7.0 的三级数据分类，不做改动。

### 8.1 Class A：生产硬决策数据

可直接进入 `Risk Controller` 和 Tier-0 硬门槛：

- `Close / OHLCV`
- `credit_spread`
- `credit_acceleration`
- `net_liquidity`
- `liquidity_roc`
- `real_yield`
- `funding_stress`

### 8.2 Class B：生产软决策数据

可进入 `Deployment Controller` 或 `Execution Overlay`：

- `VIX`、`breadth`、`Fear & Greed`
- `options walls`、`gamma regime`
- `tactical_stress_score`、`capitulation_score`、`persistence_score`

### 8.3 Class C：研究占位 / 未认证数据

不得进入生产主路径，不得改变 `risk_state`，不得改变候选认证结果。

### 8.4 缺失处理

- Class A 缺失 ≥ 2 项时，系统必须降级至 `RISK_REDUCED`。
- Class B 缺失时，系统可继续运行，但需显式标注 `usable = false`。
- Class C 默认不参与运行时决策。

## 9. QQQ 与 QLD 第一性原理

### 9.1 为什么需要分开分析

QQQ 和 QLD 看起来跟踪同一个指数，但它们的风险特征本质上不同。把两者混在一个 beta 框架里直接配比，等于忽略了 QLD 在不同市场环境下的独特危害和收益模式。

### 9.2 QQQ 的逻辑与影响因子

QQQ 跟踪 Nasdaq-100 指数，是标准的被动权益工具。

| 因子类别 | 影响因子 | 作用机制 |
| :--- | :--- | :--- |
| 宏观 | 利率环境、流动性条件 | 影响估值倍数和资金流向 |
| 基本面 | 盈利增长、Forward PE、ERP | 决定长期收益驱动力 |
| 信用 | 信用利差、融资压力 | 反映系统性风险溢价 |
| 情绪 | VIX、Fear & Greed、breadth | 反映短期定价偏离 |

**核心特征**：QQQ 的风险基本等于市场风险，没有额外结构性损耗。

### 9.3 QLD 的逻辑与影响因子

QLD 是 2x 日度杠杆 ETF，它在 QQQ 的所有因子基础上，叠加了自身的结构性风险。

| 因子类别 | 影响因子 | 作用机制 |
| :--- | :--- | :--- |
| QQQ 全部因子 | （继承） | 2x 放大收益和亏损 |
| 波动率衰减 | 日内波动率水平 | 高波动环境下即使 QQQ 持平，QLD 也亏钱（路径依赖） |
| 趋势持续性 | 连续上涨/下跌天数 | 单边趋势放大收益，震荡市侵蚀收益 |
| 融资成本 | 内含 swap 利率 | QLD 内部融资成本吞噬一部分收益 |

**核心特征**：QLD 的价值在单边上涨趋势中远超 QQQ，但在高波动震荡市和持续下跌中会被路径依赖严重伤害。

### 9.4 对配置决策的影响

- QLD 只应该在**低波动、趋势明确**的环境中占有较高权重。
- 在高波动或宏观承压环境中，即使看好 QQQ，也应该把 QLD 权重降为零（因为波动率衰减会抵消甚至逆转杠杆收益）。
- 候选认证必须对包含 QLD 的组合做波动率衰减损耗的独立测算。
- 最终输出的目标 beta 是组合级的：`portfolio_beta = qqq_pct × 1.0 + qld_pct × 2.0`，现金占比越高 beta 越低。到达同一个 beta 的路径（纯 QQQ vs QQQ+QLD）在不同波动率环境下的长期表现截然不同。

## 10. 候选配置与研究认证

v8.0 继承 v7.0 的离线认证 + 运行时确定性选择模式。

### 9.1 候选注册表

版本化对象字段不变：

- `registry_version`、`generated_at`
- `candidate_id`、`allowed_risk_state`
- `qqq_pct`、`qld_pct`、`cash_pct`
- `target_effective_exposure`
- `certification_status`（`CERTIFIED` / `CONDITIONAL` / `REJECTED`）
- `research_metrics`

### 9.2 v8.0 新增：基于 `max_beta_ceiling` 过滤

运行时选择候选时，除了按 `allowed_risk_state` 过滤外，v8.0 还必须过滤掉 `target_effective_exposure > max_beta_ceiling` 的候选。这确保 Tier-0 的压制能真正传递到候选选择层。

### 9.3 研究认证口径

不变，但新增要求：

- 认证回测必须在 `RICH_TIGHTENING` 场景下单独评估候选的 MDD，确保该 regime 下的被允许候选确实安全。

## 12. 风险预算协议

继承 v7.0：

- `Warn Band`: `20%`
- `Defense Band`: `25%`
- `Hard Budget`: `30%`

规则不变，但 v8.0 补充一条：

> 当 Tier-0 判定为 `RICH_TIGHTENING` 时，`Defense Band` 生效压制作为额外安全层——即使组合回撤尚未触及 25%，也要将 beta 上限降至 0.30，防止在高成本环境下继续扩大敞口。

## 13. 研究口径与生产口径一致性要求

v8.0 继承 v7.0 的全部一致性要求（§13），补充以下新条目：

### 13.1 v8.0 新增强一致项

- Tier-0 投影规则在研究认证和生产运行时必须完全一致。
- QQQ 和 QLD 的风险因子在研究认证和生产中使用同一套定义。

### 13.2 v8.0 新增禁止差异项

- 禁止研究中使用 Tier-0 投影但生产中跳过。
- 禁止研究中对 QLD 考虑波动率衰减但生产中忽略。

3. `AllocationState` 枚举保留（供历史数据回放），但不再作为候选选择的输入。

## 17. 验收标准

### 从 v7.0 继承

- `AC-1`: 全零或非法仓位输入时，现状快照必须回退为 `100% Cash`。
- `AC-2`: `Risk Controller` 与 `Deployment Controller` 必须分离，输出字段不可复用同一状态枚举。
- `AC-3`: 运行时不得调用全候选嵌套滚动回测来决定单日生产输出。
- `AC-4`: 生产候选必须来自版本化 `CertifiedCandidateRegistry`。
- `AC-5`: 所有生产硬决策只允许使用 Class A 数据。
- `AC-6`: Class C 数据缺失或移除时，生产决策结果不得改变。
- `AC-7`: 再平衡执行必须由"状态切换或偏离带宽触发"，不得无条件每日全量调仓。
- `AC-8`: 研究认证必须输出 `max_drawdown`、`expected_shortfall`、`mean_interval_beta_deviation`、`turnover`、`nav_integrity`。
- `AC-9`: 任一生产日必须可回放其输入、候选过滤过程和最终动作。
- `AC-10`: 危机窗口压力测试不得只展示全样本均值，必须单独展示分段结果。
- `AC-11`: 文档、报告与代码中的 AC 结论不得互相冲突。
- `AC-12`: 生产与研究使用的风险预算定义必须完全一致。

### v8.0 新增

- `AC-13`: 当 `tier0_regime = "CRISIS"` 时，`decide_risk_state()` 必须返回 `RISK_EXIT`，`target_exposure_ceiling = 0.0`，不论微观指标状态如何。
- `AC-14`: 当 `tier0_regime = "RICH_TIGHTENING"` 时，`decide_risk_state()` 必须返回至少 `RISK_REDUCED`，`target_exposure_ceiling <= 0.30`。
- `AC-15`: 当 `tier0_regime = "RICH_TIGHTENING"` 且 `capitulation_score` 达标时，`Deployment Controller` 允许输出 `DEPLOY_BASE`（不被硬锁在 `DEPLOY_SLOW`），保留左侧入场机会。
- `AC-16`: 当 `available_new_cash = 0` 时，`Deployment Controller` 必须输出 `DEPLOY_IDLE`，而非 `DEPLOY_FAST/BASE/SLOW + $0`。
- `AC-17`: 系统不得输出具体调仓金额或基于用户账户余额的计算结果。输出仅限于目标 beta、推荐配比和部署节奏。
- `AC-18`: Tier-0 的投影规则在研究认证与生产运行时必须完全一致。
- `AC-19`: QQQ 和 QLD 的风险因子必须分别分析，但最终输出的目标 beta 必须是组合级（`qqq_pct × 1.0 + qld_pct × 2.0`，现金通过占据配比空间降低整体 beta）。

## 18. 非功能要求

继承 v7.0：

- `NFR-1 Determinism`: 同一输入快照必须得到同一输出。
- `NFR-2 Latency`: 单次生产决策应在低秒级完成。
- `NFR-3 Reproducibility`: 研究结果必须可重复。
- `NFR-4 Explainability`: 每个状态切换必须有结构化证据。
- `NFR-5 Environment Safety`: 受限执行环境下不得因并行策略失效而显著退化。

## 19. 迁移原则

v8.0 迁移优先级：

1. **Tier-0 接入 Risk Controller**：修改签名，注入硬约束（最高优先，改动最小但影响最大）。
2. **Tier-0 软约束接入 Deployment Controller**：降速但保留左侧入场窗口。
3. **移除金额计算**：系统只输出目标 beta 和部署节奏，不算钱。
4. **Deployment 新增 `DEPLOY_IDLE`**：替代 `DEPLOY_FAST + $0` 的语义歧义。
5. **切断 v6 `AllocationState` 依赖**：`allocation_search.py` 接口改造。
6. **回测验证**：确认 Tier-0 软约束在历史回测中对左侧入场的保留效果。

## 20. 最终设计决策

- v8.0 的核心改造不是新增能力，而是**理清系统的身份和边界**。
- 系统是推荐引擎，不是钱包管理器。它只说"推荐 beta 是多少"和"增量资金什么时候进"。
- Tier-0 对存量 beta 是硬约束（不可商量），对增量入场是软约束（降速但不关门，给左侧入场留窗口）。
- QQQ 和 QLD 的风险因子分开分析，但输出的是一个组合级目标 beta。

这份 SRD 的核心要求只有一句话：

> 只推荐 beta 和入场节奏，不管钱，不下单。宏观压制存量但不掐死增量的左侧入场机会。
