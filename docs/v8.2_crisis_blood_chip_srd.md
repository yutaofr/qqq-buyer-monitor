# SRD: v8.2 CRISIS 血筹码战术现金回补

> 版本: v8.2
> 状态: Draft
> 日期: 2026-03-26
> 适用范围: `qqq-monitor` 的增量现金部署链路、回测与报告
> 前置文档: `docs/v6.3_full_backtest_report.md`、`docs/v8.0_linear_pipeline_srd.md`、`docs/v8.1_beta_advisory_friction_srd.md`

## 1. 文档目的

本 SRD 定义一项新的 v8.2 能力: 在 `tier0_regime == CRISIS` 的极端危机阶段，为新增现金保留一个经过严格组合因子确认的 “blood-chip” 战术回补窗口。

该能力解决 v6.3 报告中提出的 `CASH_FLIGHT` 退出过慢问题，但不改变 v8.1 对存量 beta 的防守边界。

## 2. 第一性原理

- 存量资产防守和增量现金部署必须继续分离。
- `CRISIS` 期间，系统仍应先保命，再谈抄底。
- 真正允许在危机中动用战术现金的，不应是单一超跌信号，而应是 “深坑 + 流动性/情绪/微观结构确认” 的非线性组合。
- 危机中的部署例外必须可审计，不能退化成 “任何 CRISIS 都可以买”。

## 3. 产品目标

- 保留 `CRISIS` 下存量 beta 的 `RISK_EXIT / 0.5 beta floor` 防守。
- 允许新增现金在危机反转初期进行有限、快速、可审计的回补。
- 把危机审计从 “零部署” 升级为 “零未经授权部署”。
- 在回测、验收报告和图表中显式展示危机回补是否发生以及是否合规。

## 4. 范围

### 包含

- `Deployment Controller` 的 `CRISIS` 特例通道。
- live 路径把 `price_vix`、`price_mfi`、`short_squeeze_potential`、`bond_vol_spike`、`POC` 等软信号送入 `FeatureSnapshot`。
- backtest / expectation matrix / acceptance report 对该例外通道的审计。
- 文档、回测报告、图表和指标口径更新。

### 不包含

- 不修改 `Risk Controller` 的 `CRISIS -> RISK_EXIT` 语义。
- 不修改候选搜索或 beta floor 选择逻辑。
- 不新增真实账户管理或自动下单能力。
- 不把 Class C 研究信号升级为存量 beta 的硬决策输入。

## 5. 功能需求

### 5.1 存量 beta 边界保持不变

- 当 `tier0_regime == "CRISIS"` 时，`Risk Controller` 仍必须输出 `RISK_EXIT`。
- `target_exposure_ceiling` 仍必须维持 `0.50`。
- `selected_candidate_id` 仍可落到 `exit-floor-001` 或等价的 beta floor 候选。

### 5.2 Blood-Chip Override 只作用于新增现金

- 血筹码通道只能改变 `Deployment Controller` 的 `deployment_state`。
- 该通道不能抬高 `raw_target_beta`、`advised_target_beta`、`risk_state` 或 candidate ceiling。
- 该通道的输出语义是 “允许新增现金以 `DEPLOY_FAST` 节奏回补”，不是 “解除危机”。

### 5.3 Blood-Chip Core Preconditions

以下前置条件必须同时满足:

- `tier0_regime == "CRISIS"`
- `risk_state == "RISK_EXIT"`
- `rolling_drawdown >= 0.15`
- `tactical_stress_score < 70`

最后一条用于避免把危机中的短线暴力反弹误判为底部确认后的追涨。

### 5.4 Blood-Chip Factor Paths

在满足核心前置条件后，以下任一路径成立即可触发 override:

#### 路径 A: 流动性反转

- `liquidity_roc > 0.5`
- `credit_acceleration <= 0.0`
- `twenty_day_return <= -0.08`

#### 路径 B: 极致恐慌耗尽

- `rolling_drawdown >= 0.20`
- `capitulation_score >= 30`
- 且以下至少一项为真:
  - `price_vix_divergence`
  - `short_squeeze_potential`
  - `bond_vol_spike`

#### 路径 C: 聪明钱支撑

- `rolling_drawdown >= 0.18`
- `price_mfi_divergence`
- `near_volume_poc`

### 5.5 决策输出

- 一旦 blood-chip override 生效，`deployment_state` 必须为 `DEPLOY_FAST`。
- `pause_new_cash` 必须为 `False`。
- `reasons` 中必须包含 `blood_chip_crisis_override`。
- 系统必须能区分:
  - `CRISIS blood-chip override`
  - `CRISIS unauthorized breach`

### 5.6 缺失处理

- live 路径中，`price_vix_divergence`、`price_mfi_divergence`、`short_squeeze_potential`、`bond_vol_spike`、`near_volume_poc` 缺失时默认按 `False` 处理。
- 缺失这些字段不能阻断系统运行。
- 缺失这些字段不能放宽 override 条件。

## 6. 回测与报告要求

- 回测必须记录 `blood_chip_override_active`。
- 接受度报告必须同时输出:
  - `CRISIS blood-chip overrides`
  - `CRISIS unauthorized breaches`
- `docs/backtest_report.md` 必须将危机审计语义更新为 “未经授权 breach = 0”。
- DCA timing 图和 beta recommendation 图必须重新生成，以反映最新回测结果。

## 7. 非功能要求

- 审计优先: 每一个危机期加速部署都必须能被理由链解释。
- 架构隔离: override 不能反向污染 Risk Controller。
- 回归稳定: 全量回测的 `MDD <= 30%` 目标不能被破坏。
- 向后兼容: 非 `CRISIS` 场景下现有行为不应变化。

## 8. 验收标准

- AC-1: `CRISIS` 下 `Risk Controller` 仍输出 `RISK_EXIT`，beta ceiling 仍为 `0.50`。
- AC-2: 在满足路径 A 的危机反转夹具中，`Deployment Controller` 必须输出 `DEPLOY_FAST`。
- AC-3: 在满足路径 B 的危机耗尽夹具中，`Deployment Controller` 必须输出 `DEPLOY_FAST`。
- AC-4: 在满足路径 C 的微观结构夹具中，`Deployment Controller` 必须输出 `DEPLOY_FAST`。
- AC-5: 当 `tactical_stress_score >= 70` 时，即使其余条件成立，也不得触发 blood-chip override。
- AC-6: 非 `CRISIS` 场景下，原有 `RICH_TIGHTENING` / `TRANSITION_STRESS` 左侧窗口行为不变。
- AC-7: backtest 日序列必须携带 `blood_chip_override_active`。
- AC-8: full-sample 回测中 `CRISIS unauthorized breaches = 0`。
- AC-9: full-sample 回测仍满足 `MDD <= 30%` 且测试全绿。
- AC-10: SRD / ADD / SPT / 实现计划 / 回测报告 / 图表全部同步更新。

## 9. 风险

- 阈值过松会把危机中的一般超跌误放行为 “血筹码”。
- 阈值过紧会让 override 永远不触发，功能名存实亡。
- live 专属因子与回测可得因子不一致，容易造成 “live 能触发、backtest 不可验证”。
- 如果仍沿用 `CRISIS deployment breaches = 0` 旧口径，新的设计会被自己的审计体系误判为失败。
