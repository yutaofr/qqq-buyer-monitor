## 1. 目标

这份报告单独验证增量资金的进场节奏，不使用 `CAGR` 作为评价主轴。

我们要回答的问题不是：

- “组合最后涨了多少”

而是：

- “系统在每一个资金可入场检查点，是否按 Architect 10.0 预期的节奏部署增量资金”

## 2. 方法论

### 2.1 闭环对象

新增闭环只比较增量资金节奏决策：

- `actual_pacing_t = deployment_multiplier_t`
- `expected_pacing_t = expected_deployment_multiplier_t`
- `actual_cash_t = available_new_cash_t * actual_pacing_t`
- `expected_cash_t = available_new_cash_t * expected_pacing_t`

其中：

- `available_new_cash_t` 只在 weekly funding checkpoint 上大于 `0`
- 非资金进场日不纳入节奏误差统计

### 2.2 期望节奏面

期望节奏面由独立 research surface 生成，而不是复用生产决策树。它基于：

- `credit_spread`
- `credit_accel`
- `liquidity_roc`
- `funding_stress`
- `rolling_drawdown`
- `five_day_return`
- `twenty_day_return`

输出：

- `expected_deployment_state`
- `expected_deployment_multiplier`
- `expected_deployment_cash`

### 2.3 统计评价

主指标：

- `Mean Error`
- `MAE`
- `RMSE`
- `Error Variance`
- `Error Std Dev`
- `Correlation`
- `Explained Variance`
- `Within Tolerance Ratio`
- `Cash MAE`
- `Cash RMSE`

说明：

- `Within Tolerance Ratio` 当前使用 `|error| <= 0.25`
- `Error Variance` 采用总体方差 `ddof=0`
- `Correlation / Explained Variance` 用来确认节奏面不是偶然重合

### 2.4 为什么不看 CAGR

`CAGR` 适合评价持仓路径的长期收益，不适合评价“每一笔新增资金何时进入”。

增量资金节奏是一个条件化执行系统，核心是：

- 在坏周期里是否少犯错
- 在允许部署时是否部署到位
- 是否和预期 cadence surface 一致

所以它必须有独立的 pacing audit，而不是被总收益掩盖。

## 3. 最新结果

运行命令：

```bash
MPLCONFIGDIR=/tmp/mpl python scripts/run_deployment_pacing_report.py \
  --save-dir artifacts/v10_pacing_acceptance \
  --docs-image-path docs/images/v10.0_deployment_pacing_backtest.png
```

正式结果：

- `compared_points = 1360`
- `mean_error = -0.0007`
- `MAE = 0.0007`
- `RMSE = 0.0192`
- `error_variance = 0.000367`
- `error_std_dev = 0.0192`
- `correlation = 0.9993`
- `explained_variance = 0.9985`
- `within_tolerance_ratio = 99.85%`
- `actual_mean_pacing = 0.6790`
- `expected_mean_pacing = 0.6798`
- `cash_MAE = 0.03`
- `cash_RMSE = 0.82`
- categorical overlay:
  - `deployment exact match = 99.85%`
  - `deployment within one step = 100.00%`

结论：

- v10.0 的增量资金节奏决策面和独立期望节奏面几乎重合
- 偏差不是系统性的高配或低配
- 节奏闭环已经足够强，可以作为正式 acceptance 证据

## 6. Release Implication

对 v10.0 来说，这个新闭环意味着：

- 部署节奏不再只是分类信号对齐
- 增量资金决策已经升级为连续统计可验证对象
- release 验收可以同时要求：
  - categorical deployment alignment
  - continuous pacing alignment
  - cash pacing error control

这使得 “incremental cash deployment” 成为一个完整、可审计、可回归的正式 backtest surface。
