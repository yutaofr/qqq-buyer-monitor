# SRD: v7.0 工业生产级资产配置与增量资金双控制器系统

> 版本: v7.0
> 状态: Draft
> 日期: 2026-03-24
> 适用范围: `qqq-monitor` 生产重构

## 1. 文档目的

本 SRD 定义 `qqq-monitor` v7.0 的工业生产级目标系统。

v7.0 不再把“风险管理 / 再平衡决策”和“增量资金买入时机决策”混在一个状态机里，而是显式拆分为两个控制器：

- `Risk Controller`: 决定组合目标风险暴露、现金缓冲和再平衡动作。
- `Deployment Controller`: 决定新增现金的部署速度、暂停条件和恢复节奏。

本 SRD 面向 Claude Code 直接落地，要求：

- 需求可测试，不写模糊口号。
- 数据口径、研究口径、生产口径必须一致或显式区分。
- 每条决策都必须可回放、可解释、可审计。

## 2. 背景与问题定义

v6.x 的主要问题不是“没有逻辑”，而是“逻辑层次不干净”：

- 研究回测与生产实时路径不等价。
- 每日风险校准与增量资金部署共用一个执行器，导致周转和含义混淆。
- 部分宏观 / 基本面因子仍为占位值，削弱统计可信度。
- 压力测试、总回测报告和实时选择器之间存在口径冲突。
- 运行时仍依赖嵌套 mini-backtest，复杂度高，环境敏感。

v7.0 的任务不是继续堆因子，而是先建立工业级控制面。

## 3. 产品目标

### 3.1 主目标

- 在可承受风险预算内维持长期权益复利。
- 在风险恶化时降低目标暴露，而不是靠主观叙事拖延。
- 在新增现金部署上区分“应不应该持有风险”和“应不应该加快投入”。
- 用统一证据链支持 live、backtest、stress test、audit。

### 3.2 量化目标

- `G1`: 组合级最大回撤预算长期控制在设计阈值内，默认预算 `30%`。
- `G2`: 生产决策与研究认证使用同一候选集合和同一风险认证口径。
- `G3`: 风险控制与增量部署分离后，周转率显著低于 v6.x 的每日全量再平衡模式。
- `G4`: 所有生产决策可通过 JSON / SQLite / 报告完整回放。

## 4. 非目标

- 不追求日内交易。
- 不追求市场中性。
- 不追求“最准抄底点”叙事。
- 不允许使用不可验证的模拟因子来包装收益来源。
- 不允许在运行时临时发明新候选权重。

## 5. 设计原则

- `P1 单一事实源`: 研究认证、运行时选仓、回测报告必须共享同一候选注册表和同一指标定义。
- `P2 控制器分离`: 风险暴露控制和新增现金部署是两个不同问题，必须分开建模。
- `P3 硬约束优先`: 风险预算、数据可用性、候选合法性优先于收益排序。
- `P4 无伪造数据`: 缺失即缺失，占位值不得进入生产决策主路径。
- `P5 低频执行`: 允许高频审计，不允许无条件高频调仓。
- `P6 可回放`: 任一日的结果都必须能复建输入、状态、候选、筛选和最终输出。

## 6. 系统边界

### 6.1 输入

- 市场价格与历史 OHLCV
- 宏观与流动性数据
- 信贷与融资压力数据
- 市场结构与情绪数据
- 当前组合现状
- 新增现金流信息

### 6.2 输出

- `risk_state`
- `deployment_state`
- `target_allocation`
- `rebalance_action`
- `deployment_action`
- `evidence_trace`
- `data_quality`
- `candidate_selection_audit`

## 7. v7.0 核心架构

### 7.1 四层结构

1. `Feature Plane`
   - 负责采集、对齐、质量标记、缺失处理。

2. `Research Certification Plane`
   - 离线认证候选权重是否满足长期风险预算与危机场景要求。
   - 输出版本化 `CertifiedCandidateRegistry`。

3. `Runtime Decision Plane`
   - 每日运行。
   - 不做嵌套全候选滚动回测。
   - 只在已认证候选内做确定性选择。

4. `Audit Plane`
   - 保存输入快照、状态、候选过滤结果、最终动作和执行前后暴露。

### 7.2 双控制器职责

#### A. Risk Controller

负责回答：

- 当前应持有什么级别的总风险暴露？
- 是否需要再平衡？
- 现金缓冲应提高还是降低？

只允许输出：

- `RISK_ON`
- `RISK_NEUTRAL`
- `RISK_REDUCED`
- `RISK_DEFENSE`
- `RISK_EXIT`

#### B. Deployment Controller

负责回答：

- 新增现金应该按基准速度投入、加速投入、减速投入，还是暂停？
- 如果市场从高压恢复，新增资金如何恢复正常节奏？

只允许输出：

- `DEPLOY_BASE`
- `DEPLOY_SLOW`
- `DEPLOY_FAST`
- `DEPLOY_PAUSE`
- `DEPLOY_RECOVER`

### 7.3 控制器优先级

- `Risk Controller` 是上位控制器。
- `Deployment Controller` 不得突破 `Risk Controller` 给定的风险上限。
- 若 `Risk Controller` 输出 `RISK_DEFENSE` 或 `RISK_EXIT`，则 `Deployment Controller` 最高只能输出 `DEPLOY_BASE`，通常应退化为 `DEPLOY_SLOW` 或 `DEPLOY_PAUSE`。

## 8. 数据分级与使用协议

### 8.1 数据级别

#### Class A: 生产硬决策数据

可直接进入 `Risk Controller` 硬门槛：

- `Close / OHLCV`
- `credit_spread`
- `credit_acceleration`
- `net_liquidity`
- `liquidity_roc`
- `real_yield`
- `funding_stress`

要求：

- 有真实历史。
- 有可验证来源。
- live 与 backtest 均可一致使用。

#### Class B: 生产软决策数据

可进入 `Deployment Controller` 或 `Execution Overlay`：

- `VIX`
- `breadth`
- `Fear & Greed`
- `options walls`
- `gamma regime`

要求：

- 若缺历史，不得在研究回测中伪造。
- 若仅 live 可用，则只能作为运行时软修正，不得成为长期收益结论的主要来源。

#### Class C: 研究占位 / 未认证数据

包括但不限于：

- 固定常数代理
- 无法验证的合成序列
- 仅用于展示、无真实历史的特征

规则：

- 不得进入生产主路径。
- 不得改变 `risk_state`。
- 不得改变候选认证结果。

### 8.2 缺失处理

- Class A 缺失时，系统必须降级并保守输出。
- Class B 缺失时，系统可继续运行，但需显式标注 `usable = false`。
- Class C 默认不参与运行时决策。

## 9. 候选配置与研究认证

### 9.1 候选配置原则

v7.0 保留 `QQQ / QLD / Cash` 三资产框架，但候选权重不再由运行时任意试算。

运行时只能从离线认证后的候选集中选取。

### 9.2 候选注册表

必须产出版本化对象：

- `registry_version`
- `generated_at`
- `candidate_id`
- `allowed_risk_state`
- `qqq_pct`
- `qld_pct`
- `cash_pct`
- `target_exposure`
- `stress_certification`
- `turnover_profile`
- `research_metrics`

### 9.3 研究认证口径

离线研究必须至少包含：

- 全样本回测
- 危机分段回测
- 滚动窗口回测
- 样本外窗口复核

每个候选必须输出：

- `max_drawdown`
- `cagr`
- `ulcer_index`
- `expected_shortfall`
- `mean_interval_beta_deviation`
- `turnover`
- `defense_coverage`
- `nav_integrity`

### 9.4 研究认证结论

候选只能处于以下三种状态之一：

- `CERTIFIED`
- `CONDITIONAL`
- `REJECTED`

运行时默认只允许 `CERTIFIED`。
是否允许 `CONDITIONAL`，必须通过显式配置打开。

## 10. 运行时决策模型

### 10.1 Risk Controller 输入

- `credit_spread`
- `credit_acceleration`
- `liquidity_roc`
- `funding_stress`
- `equity_risk_premium`
- 当前组合暴露
- 当前组合滚动回撤

### 10.2 Risk Controller 输出规则

#### `RISK_ON`

- 宏观未见信用扩张恶化。
- 流动性未明显收缩。
- 组合回撤低于警戒带。

#### `RISK_NEUTRAL`

- 宏观中性。
- 估值不支持进攻性加杠杆。
- 保持常规权益暴露。

#### `RISK_REDUCED`

- 信用或流动性出现单边恶化。
- 或组合回撤进入警戒带。

#### `RISK_DEFENSE`

- 信用、流动性、融资压力至少两项共振恶化。
- 或组合回撤触及强防御带。

#### `RISK_EXIT`

- 三重共振恶化。
- 或组合风险预算被突破。
- 或认证候选全部失效。

### 10.3 Deployment Controller 输入

- `tactical_stress_score`
- `capitulation_score`
- `persistence_score`
- 价格与长期均值偏离
- 市场结构确认
- 新增现金余额
- 当前 `risk_state`

### 10.4 Deployment Controller 输出规则

#### `DEPLOY_BASE`

- 正常 DCA。

#### `DEPLOY_SLOW`

- 市场承压但未形成高质量超跌。
- 或 `risk_state` 已偏保守。

#### `DEPLOY_FAST`

- 仅在 `RISK_ON` 或 `RISK_NEUTRAL` 下允许。
- 必须满足恐慌 / 超跌 / 结构确认组合条件。

#### `DEPLOY_PAUSE`

- 价格追高。
- 风险结构恶化。
- 或市场处于高不确定状态。

#### `DEPLOY_RECOVER`

- 从防御或暂停状态恢复。
- 使用平滑恢复节奏，不允许一次性跳回激进部署。

## 11. 再平衡与执行协议

### 11.1 审计频率

- 每日审计。

### 11.2 执行触发

仅在以下条件之一满足时执行再平衡：

- `risk_state` 变化
- 当前暴露偏离目标暴露超过带宽
- 现金比例偏离目标超过带宽
- 风险预算触发硬门槛

### 11.3 禁止行为

- 禁止无条件每日全量调回目标权重。
- 禁止为了部署新增现金而先卖出风险资产再买回。
- 禁止用 `effective_exposure` 冒充真实 beta。

### 11.4 暴露定义

- `effective_exposure = qqq_pct + 2 * qld_pct`
- `realized_beta` 必须基于收益序列协方差单独审计。

## 12. 风险预算协议

默认个人风险预算：

- `Warn Band`: `20%`
- `Defense Band`: `25%`
- `Hard Budget`: `30%`

规则：

- 触及 `20%` 后不得继续提升目标风险上限。
- 触及 `25%` 后必须至少进入 `RISK_REDUCED`。
- 触及 `30%` 后必须进入 `RISK_EXIT` 或同等级资本保护模式。

v7.0 不接受“报告中说守住 30%，live 只看近两年窗口”的双口径。

## 13. 研究口径与生产口径一致性要求

### 13.1 强一致项

以下内容必须一致：

- 候选集合
- 目标暴露定义
- 风险预算定义
- 回撤定义
- beta 审计定义
- 候选合法性定义

### 13.2 允许差异项

以下内容可不同，但必须显式声明：

- 研究使用全样本与多危机场景
- 运行时只从已认证候选中选择
- 运行时可使用短期 overlay 作为部署节奏修正

### 13.3 禁止差异项

- 禁止研究使用全历史风险认证，生产却用短窗口替代同一硬约束。
- 禁止研究使用真实宏观状态，生产评分却退化为价格回撤近似。

### 14.1 新核心模型

## 16. 验收标准

- `AC-1`: 全零或非法仓位输入时，现状快照必须回退为 `100% Cash`。
- `AC-2`: `Risk Controller` 与 `Deployment Controller` 必须分离，输出字段不可复用同一状态枚举。
- `AC-3`: 运行时不得调用全候选嵌套滚动回测来决定单日生产输出。
- `AC-4`: 生产候选必须来自版本化 `CertifiedCandidateRegistry`。
- `AC-5`: 所有生产硬决策只允许使用 Class A 数据。
- `AC-6`: Class C 数据缺失或移除时，生产决策结果不得改变。
- `AC-7`: 再平衡执行必须由“状态切换或偏离带宽触发”，不得无条件每日全量调仓。
- `AC-8`: 研究认证必须输出 `max_drawdown`、`expected_shortfall`、`mean_interval_beta_deviation`、`turnover`、`nav_integrity`。
- `AC-9`: 任一生产日必须可回放其输入、候选过滤过程和最终动作。
- `AC-10`: 危机窗口压力测试不得只展示全样本均值，必须单独展示分段结果。
- `AC-11`: 文档、报告与代码中的 AC 结论不得互相冲突；若冲突，以最新审计结果为准并触发文档更新。
- `AC-12`: 生产与研究使用的风险预算定义必须完全一致。

## 17. 非功能要求

- `NFR-1 Determinism`: 同一输入快照必须得到同一输出。
- `NFR-2 Latency`: 单次生产决策应在低秒级完成，不依赖重型嵌套回测。
- `NFR-3 Reproducibility`: 研究结果必须可重复。
- `NFR-4 Explainability`: 每个状态切换必须有结构化证据。
- `NFR-5 Environment Safety`: 在受限执行环境下不得因并行策略失效而显著退化或报错。

## 18. 迁移原则

v7.0 迁移必须按以下顺序进行：

1. 先建立新数据模型与双控制器枚举。
2. 再建立离线候选认证与 registry。
3. 再替换运行时选择器，禁止 live mini-backtest。
4. 再替换执行层为“带宽触发再平衡”。
5. 最后重写报告与回放。

## 19. 最终设计决策

- v7.0 的主产品不是“买点信号机”，而是“风险暴露控制器 + 增量资金部署器”。
- v7.0 的生产核心不是运行时滚动搜索，而是“离线认证 + 运行时确定性选择”。
- v7.0 的可信度不来自更多因子，而来自更少但真实、可回放、可验证的因子。

这份 SRD 的核心要求只有一句话：

> 先把控制面做对，再谈收益优化。
