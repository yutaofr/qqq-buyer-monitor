# Specification and Planning Table (SPT) - Web Distribution v8.2

## 1. 核心目标 (Core Objective)
构建一个具备“自愈感知”能力的、跨越物理时区与交易日历的、零耦合且具备 Alpha 保护的生产级发布流水线。

---

## 2. 用户故事与验收标准 (User Stories & AC)

### User Story 1: 非线性失效逻辑 (The Market Pulse)
**作为一名投资者**，我希望系统能够准确识别周五收盘、半天交易日及节假日的边界，以便在休市期间看到有效的“冷冻快照”，而不是虚假的红色警报。

| 任务编号 | 任务描述 (Tasks) | 验收标准 (Acceptance Criteria) |
| :--- | :--- | :--- |
| **T1.1** | 集成 `pandas_market_calendars` 并封装 `MarketCursor` 类。 | 能够识别 NYSE 1:00 PM (半天) 与 4:00 PM (全天) 的精确收盘时刻。 |
| **T1.2** | 实现 `get_expires_at_utc(now)` 函数。 | **AC-1.1**: 输入周五 16:01 EST，输出下周一 09:30 EST + 4h 的 UTC 时间。 |
| **T1.3** | 实现 `market_state` 三态判定逻辑 (`ACTIVE | FROZEN | STALE`)。 | **AC-1.2**: 交易时间内输出 `ACTIVE`；收盘后且未过有效期输出 `FROZEN`。 |
| **T1.4** | 强制执行 Aware Datetime 与 ISO-8601 (Z) 序列化。 | **AC-1.3**: JSON 字段必须携带 `Z` 后缀，禁止出现本地时区偏移。 |

### User Story 2: 环境栅栏与原子分发 (Atomic Gating)
**作为一名系统管理员**，我希望本地测试与生产发布在物理上绝对隔离，且生产流水线具备并发互斥能力，以防止脏写和环境污染。

| 任务编号 | 任务描述 (Tasks) | 验收标准 (Acceptance Criteria) |
| :--- | :--- | :--- |
| **T2.1** | 编写 `WebExporter` 的环境感知逻辑。 | **AC-2.1**: 仅当 `GITHUB_ACTIONS=true` 且持有生产 Token 时允许 PUT 生产桶。 |
| **T2.2** | 配置 GitHub Actions 并发锁 (`concurrency`)。 | **AC-2.2**: 同时触发两个 Action 时，旧任务必须被立即 `canceled`，保留最新任务。 |
| **T2.3** | 实现 Vercel Blob 的原子覆写与 Cache-Control 注入。 | **AC-2.3**: PUT 请求必须包含 `s-maxage=3600, max-age=0`，且成功覆盖旧 `status.json`。 |

### User Story 3: “失效即死亡”防御性前端 (Fail-to-Stale UI)
**作为一名公众查询用户**，我希望在数据链路断裂或过期时看到明确的警告，而不是过时的“伪实时”数据。

| 任务编号 | 任务描述 (Tasks) | 验收标准 (Acceptance Criteria) |
| :--- | :--- | :--- |
| **T3.1** | 编写 Vanilla JS `fetch` 逻辑，带严酷 `try/catch`。 | **AC-3.1**: 网络 4xx/5xx 或 JSON 格式损坏时，UI 必须 100% 进入红色警报态。 |
| **T3.2** | 实现前端物理失效校验逻辑。 | **AC-3.2**: `if (new Date() > expires_at_utc)` 必须触发全屏 `grayscale(1)` 滤镜锁定。 |
| **T3.3** | UI 状态自适应渲染。 | **AC-3.3**: 根据 `market_state` 自动切换背景色（绿/蓝/红），文字说明必须准确。 |

### User Story 4: 带外主动告警 (The Heartbeat)
**作为一名开发者**，我希望在发布流水线崩溃的第一时间收到通知，而不是等到用户投诉。

| 任务编号 | 任务描述 (Tasks) | 验收标准 (Acceptance Criteria) |
| :--- | :--- | :--- |
| **T4.1** | 配置 GitHub Actions `if: failure()` 告警钩子。 | **AC-4.1**: 故意使脚本抛出异常，Webhook 必须在 5 分钟内收到 Run ID 消息。 |
| **T4.2** | 注入数据完整性元数据 (`integrity_level`)。 | **AC-4.2**: 信号采集源若低于阈值，`fidelity_level` 必须在 JSON 中下迁至 `DEGRADED`。 |

## 3. 验收演练路径 (Test-Driven Roadmap)

1.  **阶段一：极限界限测试 (The Crucible)**: 运行 `pytest tests/unit/test_web_exporter.py`，覆盖夏令时、周五收盘、半天交易。
2.  **阶段二：影子发布测试 (Shadow Deployment)**: 本地 `EXPORT_WEB=1` 运行，验证 `dev-status.json` 格式。
3.  **阶段三：生产冒烟测试 (Live Smoke Test)**: 触发 GA，F12 检查 `status.json` 的 `Cache-Control` 头。
