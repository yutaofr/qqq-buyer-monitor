## 1. 架构目标

本次设计要解决的问题不是 “解除危机”，而是 “在危机仍未解除时，为新增现金提供受控回补通道”。

因此架构上明确采用以下边界:

1. `Risk Controller` 不动。
2. `Deployment Controller` 在 `CRISIS` 下新增一条可审计的 override 分支。
3. 回测审计从 “危机零部署” 切换到 “危机零未经授权部署”。

## 2. 设计原则

- 防守优先: 存量 beta 防守语义不变。
- 局部例外: 只给新增现金开口子。
- 组合确认: 必须是多因子非线性组合，不允许单点阈值触发。
- 回测可证: 至少保留一条只依赖现有 backtest 因子的核心路径。
- 审计可追溯: 每一次 override 都必须有布尔标记和理由链。

## 3. 模块设计

### 3.1 `src/engine/deployment_controller.py`

新增两个私有 helper:

- `_build_blood_chip_context(...)`
- `_detect_blood_chip_crisis_override(...)`

职责:

- 解析 `rolling_drawdown`、`credit_acceleration`、`liquidity_roc`、`capitulation_score`、`tactical_stress_score`、`twenty_day_return` 以及 live 软特征。
- 返回 `(override_active, evidence_dict)`。

决策顺序调整为:

1. 先计算 `blood_chip_override`
2. `risk_state == RISK_EXIT` 时:
   - 若 `blood_chip_override=False` → 维持 `DEPLOY_PAUSE`
   - 若 `blood_chip_override=True` → 允许继续进入 `DEPLOY_FAST`
3. `credit_spread >= 650` 或 `rolling_drawdown >= 0.25` 的 `deep_drawdown_pause` 同样需要尊重 override 结果
4. `tactical_stress_pause` 仍高于 blood-chip override

### 3.2 `src/engine/feature_pipeline.py`

将以下软特征纳入 Class B:

- `price_vix_divergence`
- `price_mfi_divergence`
- `short_squeeze_potential`
- `bond_vol_spike`
- `near_volume_poc`

这样它们在 `FeatureSnapshot` 中具有明确的 “可进入 Deployment Controller” 语义，而不是继续漂浮在 live 旁路对象里。

### 3.3 `src/main.py`

live 路径将 Tier-1 / Tier-2 的已有结果接入 snapshot:

- `price_vix_divergence = tier1.divergence_flags["price_vix"]`
- `price_mfi_divergence = tier1.divergence_flags["price_mfi"]`
- `short_squeeze_potential = tier1.divergence_flags["short_squeeze_potential"]`
- `bond_vol_spike = tier1.divergence_flags["bond_vol_spike"]`
- `near_volume_poc = abs(price - tier2.poc) / price <= 0.02`

同时把 override 命中结果写入:

- `deployment_action["blood_chip_override_active"]`
- `logic_trace`

### 3.4 `src/backtest.py`

`build_signal_timeseries()` 增加:

- `blood_chip_override_active`
- `deployment_reason_rule`

backtest 默认仍只依赖现有 canonical macro + 价格数据。

为保证回测可证，expectation surface 中的核心 override 路径采用只依赖以下数据的 Path A:

- `credit_spread`
- `credit_accel`
- `liquidity_roc`
- `rolling_drawdown`

live 专属路径 B/C 在 backtest 中默认为 False，除非测试 fixture 显式提供。

## 4. 审计设计

### 4.1 授权与违规定义

- `blood_chip_override_active=True` 且 `tier0_regime=="CRISIS"` 且 `deployment_state!="DEPLOY_PAUSE"`:
  - 记为 authorized override
- `blood_chip_override_active=False` 且 `tier0_regime=="CRISIS"` 且 `deployment_state!="DEPLOY_PAUSE"`:
  - 记为 unauthorized breach

### 4.2 对外解释

CLI / report 需要明确告诉读者:

- 这不是 “危机解除”
- 这不是 “抬高存量 beta”
- 这只是 “允许新增现金以更快速度回补”

## 5. 风险与缓解

- 风险: 只改 `Deployment Controller` 会与旧的危机回测口径冲突。
  - 缓解: 同步修改 acceptance report 和 backtest report。
- 风险: live 特征进入 snapshot 后可能被误当作 Risk Controller 输入。
  - 缓解: 明确将其定义为 Class B。
- 风险: 危机路径过多导致逻辑不透明。
  - 缓解: helper 返回结构化 evidence，统一落入 `reasons`。

## 6. 兼容性

- 非 `CRISIS` 场景行为保持不变。
- `RiskDecision` 结构不变。
- `DeploymentDecision` 结构不变，仅扩展 `reasons` 的 rule 值。
- 现有图表脚本和报告路径保持不变，只更新数据和文案。
