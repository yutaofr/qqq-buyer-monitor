# v9.0 风险管理矩阵（四眼复核版）

> 状态: Revised after architecture feedback
> 日期: 2026-03-28
> 适用范围: `qqq-monitor` 当前主分支与下一版 SRD
> 目标: 在第三方评审基础上，重新给出一版逻辑自洽、符合第一性原理、并由仓内统计与外部研究共同支撑的风险管理矩阵。

## 1. 先给结论

这次复核后，v9.0 必须先纠正三个边界问题:

1. **系统的主 contract 是组合级 `target_beta` 信号，不是唯一资产配比。**
   当前代码里可以附带一个 `QQQ / QLD / Cash` 的实现示例，但产品边界仍然是“推荐目标 beta 和新增现金节奏”，不是替用户锁死唯一配比路径。
2. **`QLD` 不应与 `target_beta > 1.0` 绑定。**
   从第一性原理上，任何 `0 < beta <= 1.2` 都可以由多种 `QQQ / QLD / Cash` 组合实现。低 beta 不等于不能持有 `QLD`，而只意味着 `QLD` 的允许占比必须更小、现金缓冲必须更大。
3. **`Deployment Controller` 只管新增现金节奏，且其操作语义应统一解释为“是否按节奏买入 QQQ 新现金”。**
   它不参与存量 beta 计算，不授予 `QLD` 权限，也不反向抬高 `risk_state` 或 `target_beta`。

因此，v9.0 的核心表达不该是:

> “哪些因子决定 `QQQ / QLD / Cash` 的唯一配比。”

而应是:

> “哪些因子决定组合级 `target_beta` 的硬边界，哪些因子只决定新增现金买入 QQQ 的快慢，以及在给定 `target_beta` 下，`QLD` 允许占比应如何收缩。”

## 2. 第一性原理: 系统先输出 beta，再由用户决定怎么实现

### 2.1 组合级 beta 才是系统真正要负责的变量

系统的核心量是:

```text
portfolio_beta = qqq_pct * 1.0 + qld_pct * 2.0 + cash_pct * 0.0
```

只要满足:

```text
qqq_pct >= 0
qld_pct >= 0
cash_pct >= 0
qqq_pct + qld_pct + cash_pct = 1
qqq_pct + 2 * qld_pct = target_beta
```

那么同一个 `target_beta` 就有不止一种实现路径。

### 2.2 `QLD` 可以出现在任何 beta 档位

对 `beta <= 1.0`，只要 `QLD` 比例够小、`cash` 比例够高，`QLD` 完全可以参与。

例子:

| 目标 beta | 可行实现 A | 可行实现 B | 可行实现 C |
| :--- | :--- | :--- | :--- |
| `0.8` | `QQQ 80 / QLD 0 / Cash 20` | `QQQ 60 / QLD 10 / Cash 30` | `QQQ 50 / QLD 15 / Cash 35` |
| `0.9` | `QQQ 90 / QLD 0 / Cash 10` | `QQQ 70 / QLD 10 / Cash 20` | `QQQ 80 / QLD 5 / Cash 15` |
| `1.0` | `QQQ 100 / QLD 0 / Cash 0` | `QQQ 60 / QLD 20 / Cash 20` | `QQQ 50 / QLD 25 / Cash 25` |

当前仓库里的认证候选也直接证明了这一点:

| Candidate | Risk State | 配比 | 目标 beta |
| :--- | :--- | :--- | ---: |
| `defense-002` | `RISK_DEFENSE` | `50 / 15 / 35` | `0.8` |
| `neutral-base-001` | `RISK_NEUTRAL` | `70 / 10 / 20` | `0.9` |
| `neutral-low-drift` | `RISK_NEUTRAL` | `80 / 5 / 15` | `0.9` |
| `neutral-conditional-001` | `RISK_NEUTRAL` | `60 / 20 / 20` | `1.0` |

所以，v9.0 不能再写成:

- “`QLD` 只在 `target_beta > 1.0` 时才允许出现”
- “`QLD` 只能在 `RISK_ON / RISK_NEUTRAL` 存在”

更准确的表述应是:

- `QLD` 是同一底层指数的杠杆实现路径；
- `stress` 越大，允许的 **`QLD share ceiling`** 越低；
- `RISK_REDUCED` 乃至更防守的 beta 档位，仍然可以允许**有限 `QLD` 参与**，前提是总 beta、现金缓冲和回撤预算都满足约束。

### 2.3 三类资产各自扮演什么角色

| 资产 | 第一性原理角色 | 真正要管的风险 |
| :--- | :--- | :--- |
| `QQQ` | 核心非杠杆 Nasdaq beta | 估值、信用、流动性、组合回撤 |
| `QLD` | 杠杆实现路径，不是独立底层 | 路径依赖、波动拖累、强平式左尾 |
| 货币基金 / 现金等价物 | `beta ~= 0` 的缓冲层 | 流动性、短久期、等待更高赔率再部署 |

`QLD` 的关键不是“能不能存在”，而是“在当前压力状态下，最多应该占多少权重”。

## 3. 当前主分支真正已经做了什么

### 3.1 系统边界

`README-CN.md` 和 `docs/v8.0_linear_pipeline_srd.md` 对主边界写得很清楚:

- 系统只推荐 **组合级目标 beta**
- 系统只推荐 **新增现金入场节奏**
- 系统 **不计算金额**
- 系统 **不管理账户**
- 系统 **不自动执行**

因此，`QQQ / QLD / Cash` 的具体比例，在 v9.0 文档里只能被写成:

- 当前实现示例
- 当前认证候选
- 用户可选实现路径

不能被写成“唯一合同输出”。

### 3.2 Tier-0 与 Risk Controller

当前硬链路是:

- `Tier-0`: `credit_spread + ERP`
- `Risk Controller`: `credit_spread + credit_acceleration + liquidity_roc + funding_stress + rolling_drawdown`

当前不在 hard-risk 链上的包括:

- `breadth`
- `ma200_dev`
- `fear_greed`
- `price_vix_divergence`

所以，第三方文档中把 `breadth + ma200_dev` 直接写成 stock-beta veto 的部分，不能视为“当前主分支事实”，只能视为未来候选方案。

### 3.3 Deployment Controller

当前 `DeploymentDecision` 只输出:

- `deployment_state`
- `dca_multiplier`
- `pause_new_cash`
- `reasons`

它 **没有资产配比字段**。

这意味着在生产语义上，`Deployment Controller` 只是在回答:

> “现在要不要按更快 / 更慢 / 暂停的节奏部署新增现金？”

根据当前产品边界，这个“新增现金部署”应统一理解为:

- **按节奏买入 QQQ 新现金**

而不是:

- 重新计算存量 `target_beta`
- 授权增加 `QLD`
- 改写 `risk_state`

要点是:

- 新增现金不参与存量 beta 决策链；
- Deployment 只影响新现金买入时点；
- QLD 是否参与，属于用户如何实现 `target_beta` 的问题，不是 Deployment 的权限。

### 3.4 当前 candidate registry 的正确定位

当前 `candidate_registry_v7.json` 仍然有价值，但价值在于:

- 证明哪些 `QQQ / QLD / Cash` 组合已经被离线认证；
- 提供若干 `target_beta` 的可实现示例；
- 支持 backtest / runtime 的内部 materialization。

它**不应被提升为产品 contract 本身**。

v9.0 文档里如果引用候选池，正确写法应是:

- “当前 checked-in registry 提供了以下可行示例”

而不是:

- “系统要求必须输出这组配比”

## 4. 因子影响矩阵

下面的矩阵把因子按权限拆成两层:

- **Beta Hard Layer**: 决定组合级 `target_beta` 的上限和 `QLD share ceiling`
- **QQQ New-Cash Layer**: 只决定新增现金买入 QQQ 的快慢

### 4.1 Beta Hard Layer

| 因子 | 对 `target_beta` 的作用 | 对 `QLD share` 的作用 | 对 `QQQ` 的作用 | 证据等级 |
| :--- | :--- | :--- | :--- | :--- |
| `credit_spread` | 决定是否需要显著降 beta | 高 spread 时应大幅压低 `QLD share ceiling` | 仍可保留核心底仓 | `A` |
| `credit_acceleration` | 放大路径风险，推动去杠杆 | 使 `QLD` 更不适合做主要实现路径 | 对 `QQQ` 也是负面，但程度弱于 `QLD` | `A` |
| `liquidity_roc` | 流动性快速收缩时应降 beta | 对 `QLD` 更不友好 | 对 `QQQ` 负面 | `B` |
| `funding_stress` | 作为共振确认，压低 beta | 杠杆融资环境变差时应压缩 `QLD share` | 对 `QQQ` 负面 | `B` |
| `rolling_drawdown` | 组合风险预算硬约束 | 连续亏损环境下应优先减少 `QLD` | 保留 `QQQ` 底仓但降总 beta | `A` |
| `ERP / real_yield / forward_pe` | 低 ERP / 高 real yield 时压低 beta | 估值缓冲不足时不适合高 `QLD share` | 对 `QQQ` 是结构逆风 | `C` |

### 4.2 QQQ New-Cash Layer

| 因子 | 用途 | 对 `target_beta` | 对 `QLD` | 结论 | 证据等级 |
| :--- | :--- | :--- | :--- | :--- | :--- |
| `five_day_return` | 节奏确认 | 不直接改 | 不授权 | 只决定 QQQ 新现金快慢 | `A` |
| `twenty_day_return` | 节奏确认 | 不直接改 | 不授权 | 只决定 QQQ 新现金快慢 | `A` |
| `tactical_stress_score` | 暂停条件 | 不直接改 | 不授权 | 只决定 QQQ 新现金快慢 | `A` |
| `capitulation_score` | 左侧机会识别 | 不直接改 | 不授权 | 只决定 QQQ 新现金快慢 | `A` |
| `price_vix_divergence / price_mfi_divergence / short_squeeze_potential / bond_vol_spike / near_volume_poc` | `blood-chip` 例外确认 | 不直接改 | 不授权 | 只决定危机中是否允许更快买入 QQQ 新现金 | `A/B` |

### 4.3 当前不该升格为 hard veto 的因子

| 因子 | 为什么不能直接升格 |
| :--- | :--- |
| `breadth` | 当前是 Class B soft signal，不在 hard-risk 数据契约里 |
| `ma200_dev` | 当前主分支未接入 hard-risk 链 |
| `FCF yield` | live 仍是 placeholder |
| `earnings revisions breadth` | live 仍是 placeholder |

这不是否认它们有信息含量，而是说:

- 它们当前没有足够证据去决定 `QLD share ceiling`
- 更没有资格越权决定 `target_beta`

## 5. 统计 double check

### 5.1 当前信号链是稳定的

本地复核结果:

- `beta_mae = 0.0559`
- `beta_rmse = 0.1688`
- `beta_within_tolerance = 88.97%`
- `deployment_exact_match = 100%`
- `beta_floor_respected = True`
- `beta_cap_respected = True`
- `CRISIS unauthorized breaches = 0`

说明:

- 现有 hard-beta 链和 deployment 链在当前验收口径下是稳定的；
- 需要修正的是文档表述，不是当前“Deployment 越权改 beta”的代码行为。

### 5.2 当前主分支常见发出的 beta 信号

`artifacts/v9_review/signal_acceptance_summary.json` 显示，当前对齐回测里最常见的信号是:

| `target_beta` | 出现天数 |
| :--- | ---: |
| `0.5` | `3770` |
| `1.0` | `1208` |
| `1.2` | `1822` |

这说明当前 checked-in candidate set 与 advisory friction 的组合，确实把主分支压缩成了较离散的 beta 信号。

但必须强调:

- 这只是**当前实现统计结果**
- 不是“beta 只能由这三种资产配比唯一实现”的理论定律

### 5.3 为什么高 stress 下必须优先压 `QLD share`

用同一份历史数据，把 `QQQ` 与按系统方法模拟的 `QLD` 做 60 日前瞻比较:

| 信用利差区间 | 样本数 | QQQ 60d 平均收益 | QQQ 60d 平均前瞻 MDD | QLD 60d 平均收益 | QLD 60d 平均前瞻 MDD |
| :--- | ---: | ---: | ---: | ---: | ---: |
| `< 450 bps` | `3153` | `+3.26%` | `-5.01%` | `+6.05%` | `-9.91%` |
| `450-550 bps` | `1274` | `+6.73%` | `-4.90%` | `+13.05%` | `-9.83%` |
| `550-650 bps` | `747` | `+0.98%` | `-7.90%` | `+1.17%` | `-15.42%` |
| `>= 650 bps` | `1553` | `+0.69%` | `-11.67%` | `+1.24%` | `-21.63%` |

正确结论不是:

- “高 spread 下绝不能持有 `QLD`”

而是:

- 高 spread 下，`QLD` 的前瞻左尾损伤显著快于 `QQQ`
- 所以 stress 升高时，应该先压 **`QLD share ceiling`**
- 即便用户仍想用 `QLD` 来实现同一个较低 `target_beta`，也必须搭配更高现金缓冲，且不能让 `QLD` 成为主要风险来源

### 5.4 `credit_acceleration` 的正确解读

样本里 `credit_acceleration > 15%` 之后，未来 60 日平均收益并不总是差，因为危机后强反弹会拉高均值。

但这不构成“可以继续上杠杆”的证据。真正有用的信息是:

- 它会显著恶化前瞻 MDD
- 尤其会放大 `QLD` 的路径损伤

所以它更适合被解释成:

- **去杠杆放大器**

而不是:

- **单因子清仓按钮**

## 6. 对第三方评审的 adopt / modify / reject

| 第三方建议 | 处理 | 修正后的表达 |
| :--- | :--- | :--- |
| 保留 `0.5x beta floor`，强调 long-bias | `ADOPT` | 与当前系统边界一致 |
| 强化 Tier-0 对信用与估值的主导权 | `ADOPT` | 但 `ERP` 仍属于 live/theory 强、历史契约弱 |
| 把 `QLD` 视为比 `QQQ` 更敏感的风险实现路径 | `ADOPT` | 但不是只有 `beta>1.0` 才能使用 |
| 把 `credit_acceleration` 写成单因子 hard veto | `MODIFY` | 应作为共振放大器，不应单独独裁 |
| 把 `breadth + ma200_dev` 升格为 hard veto | `REJECT for now` | 当前证据链不够 |
| 把 Deployment 描述成“抬高存量 beta 的战术后门” | `REJECT` | 正确语义是“QQQ 新现金节奏后门” |
| 把认证候选配比写成系统必需输出 | `REJECT` | 正确主 contract 是 `target_beta` |

## 7. 架构师建议

### 7.1 立即坚持

- 把 **组合级 `target_beta`** 写成 v9.0 的一等公民。
- 把 **`QLD share ceiling`** 作为比 “`QLD` 是否存在” 更准确的控制变量。
- 把 Deployment 明确成 **QQQ 新现金节奏引擎**。
- 把 candidate registry 定位为 **内部认证实现路径**，不是用户 contract。

### 7.2 不要再写错的三件事

- 不要再把 `QLD` 与 `beta > 1.0` 绑定。
- 不要再把 `RISK_REDUCED` 写成天然排斥 `QLD`。
- 不要再把 Deployment 写成会抬高存量 beta 或授予 `QLD` 权限。

### 7.3 下一步研究

如果未来真的要进一步细化 v9:

1. 把 `erp / forward_pe` 补进 canonical historical contract。
2. 把 `breadth / ma200_dev` 纳入 research contract，再做离线认证。
3. 对同一 `target_beta` 下不同 `QLD share` 的长期 path risk 做系统性比较，研究“同 beta 下哪种实现路径更优”。

第 3 点很重要。因为对 v9.0 来说，真正缺的不是 “beta 怎么算”，而是:

> “在同一个 `target_beta` 下，`QQQ` 与 `QLD` 的最优拆分应该怎样随 stress 变化。”

## 8. 最终判定

本次复核后的 v9.0 正确结论是:

- 系统真正要输出的是 **组合级 `target_beta`**
- `QLD` 是该 beta 的一种杠杆实现路径，**可以出现在任何 beta 档位**
- stress 升高时，系统应该压的是 **总 beta** 和 **`QLD share ceiling`**
- Deployment 只决定 **是否按快/慢/暂停去买入 QQQ 新现金**

所以，最自洽的风险矩阵不是:

> “什么状态下 `QLD` 完全存在，什么状态下 `QLD` 完全消失。”

而是:

> “什么状态下系统把 `target_beta` 压到哪里，以及在这个 beta 下，`QLD` 最多还能占多少实现权重。”
